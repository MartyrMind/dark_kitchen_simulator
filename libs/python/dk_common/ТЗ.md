[text](tz_agent_stage_1_dk_common.md)# ТЗ для агента: Этап 1 — общая библиотека `dk-common`

## 1. Контекст

Проект — микросервисная система управления выполнением заказов для dark kitchen.
Python-сервисы проекта должны использовать общую техническую библиотеку:

```text
libs/python/dk-common
```

Библиотека нужна, чтобы не копировать базовую инфраструктуру между Python-сервисами.

Этап 1 посвящен только общей инфраструктуре:

```text
- настройки сервисов;
- логирование;
- correlation_id / request_id;
- healthcheck;
- базовые инфраструктурные ошибки.
```

На этом этапе не нужно реализовывать бизнес-метрики, сложные Mongo events, доменную логику или код конкретных сервисов.

---

## 2. Цель этапа

Реализовать installable Python package `dk-common`, который смогут подключать все Python-сервисы проекта.

После завершения этапа любой Python-сервис должен уметь:

```python
from dk_common.logging import configure_logging
from dk_common.settings import BaseServiceSettings
from dk_common.correlation import CorrelationIdMiddleware
from dk_common.health import create_health_router
```

И использовать единый подход для:

```text
- конфигурации через env-переменные;
- структурированного логирования через Loguru;
- JSON-логов в контейнерном режиме;
- readable-логов в локальном режиме;
- correlation_id/request_id middleware;
- endpoint-а GET /health;
- базовых инфраструктурных ошибок.
```

```text
1. BaseSettings для конфигурации сервисов.

- HTTP-клиенты к конкретным сервисам;

- Fulfillment Service;

- Kitchen Scheduler Worker на Go.
```

## 4. Ограничения архитектуры

`dk-common` — это только техническая библиотека.

В `dk-common` разрешено размещать:

```text
- настройку Loguru;
- JSON-форматирование логов;
- correlation_id / request_id middleware;
- базовые настройки Pydantic Settings;
- healthcheck helpers;
- общие исключения инфраструктурного уровня.
```

В `dk-common` запрещено размещать:

```text
- SQLAlchemy-модели конкретных сервисов;
- Alembic-миграции;
- бизнес-логику заказов, рецептов, станций или KDS;
- репозитории сервисов;
- use cases сервисов;
- код, который делает сервисы зависимыми от чужой внутренней модели данных.
```

Правило:

```text
Общая библиотека может знать, как логировать и конфигурировать сервис.
Общая библиотека не должна знать, что такое order, kitchen_task, station, recipe или KDS.
```

## 8. Модуль `dk_common.settings`

### Назначение

Реализовать базовый класс настроек для Python-сервисов.

### Требуемый API

```python
from dk_common.settings import BaseServiceSettings
```

### Пример использования

```python
from dk_common.settings import BaseServiceSettings

class Settings(BaseServiceSettings):
    database_url: str

settings = Settings()
```

### Обязательные поля

`BaseServiceSettings` должен содержать минимум:

```python
service_name: str = "unknown-service"
environment: str = "local"
log_level: str = "INFO"
log_format: str = "readable"
debug: bool = False
```

### Env-переменные

Настройки должны читаться из env:

```env
SERVICE_NAME=kitchen-service
ENVIRONMENT=local
LOG_LEVEL=INFO
LOG_FORMAT=readable
DEBUG=false
```

### Валидация

`LOG_FORMAT` должен поддерживать только:

```text
json
readable
```

Если задано другое значение, должна быть понятная ошибка валидации.

### Критерии приемки

```text
- BaseServiceSettings() создается с дефолтами.
- Env-переменные переопределяют дефолты.
- log_format валидируется.
- Класс можно наследовать в сервисах.
```

## 9. Модуль `dk_common.logging`

### Назначение

Реализовать единую настройку логирования через Loguru.

### Требуемый API

```python
from dk_common.logging import configure_logging, get_logger
```

### `configure_logging()`

Сигнатура:

```python
def configure_logging(
    service_name: str,
    environment: str = "local",
    log_level: str = "INFO",
    log_format: str = "readable",
) -> None:
    ...
```

### Поведение

Функция должна:

```text
1. Удалять дефолтные handlers Loguru.
2. Настраивать вывод в stdout/stderr.
3. Поддерживать log_format="readable" для локальной разработки.
4. Поддерживать log_format="json" для Docker/Kubernetes.
5. Добавлять в каждый лог обязательные поля.
6. Не создавать дублирующие handlers при повторном вызове.
```

### Обязательные поля лога

```text
timestamp
level
service
environment
message
correlation_id
request_id
```

Для `correlation_id` и `request_id`, если они не установлены, использовать `null` или `"-"`.

### JSON-режим

В `log_format="json"` лог должен быть валидной JSON-строкой.

Пример:

```json
{
  "timestamp": "2026-04-27T18:20:00Z",
  "level": "INFO",
  "service": "kitchen-service",
  "environment": "local",
  "message": "Service started",
  "correlation_id": "corr_123",
  "request_id": "req_456"
}
```

### Readable-режим

В `log_format="readable"` допустим человекочитаемый формат.

Пример:

```text
2026-04-27 18:20:00 | INFO | kitchen-service | corr=corr_123 | Service started
```

### Context support

Логирование должно уметь брать `correlation_id` и `request_id` из contextvars.

Если middleware установил correlation/request id, они должны автоматически попадать в логи внутри обработки запроса.

### `get_logger()`

Сигнатура:

```python
def get_logger():
    ...
```

Может возвращать `loguru.logger`.

### Критерии приемки

```text
- configure_logging() можно вызвать несколько раз без дублирования handlers.
- JSON-режим пишет валидный JSON.
- Readable-режим пишет человекочитаемый текст.
- В логах есть service, environment, level, timestamp, message.
- При установленном correlation context в логах есть correlation_id.
```

## 10. Модуль `dk_common.correlation`

### Назначение

Реализовать middleware для `correlation_id` и `request_id`.

### Требуемый API

```python
from dk_common.correlation import (
    CorrelationIdMiddleware,
    get_correlation_id,
    get_request_id,
    set_correlation_id,
    set_request_id,
)
```

### Headers

Middleware должен работать с HTTP headers:

```http
X-Correlation-ID
X-Request-ID
```

### Поведение

Для каждого HTTP-запроса:

```text
1. Если входящий X-Correlation-ID есть — использовать его.
2. Если входящего X-Correlation-ID нет — сгенерировать новый UUID.
3. Если входящий X-Request-ID есть — использовать его.
4. Если входящего X-Request-ID нет — сгенерировать новый UUID.
5. Сохранить оба значения в contextvars.
6. Добавить оба значения в response headers.
7. Очистить contextvars после завершения запроса.
```

### Пример подключения

```python
from fastapi import FastAPI
from dk_common.correlation import CorrelationIdMiddleware

app = FastAPI()
app.add_middleware(CorrelationIdMiddleware)
```

### Критерии приемки

```text
- Если клиент прислал X-Correlation-ID, response возвращает тот же id.
- Если клиент не прислал X-Correlation-ID, response содержит сгенерированный id.
- X-Request-ID работает аналогично.
- get_correlation_id() возвращает id во время обработки запроса.
- Значения не протекают между разными запросами.
- Middleware совместим с FastAPI.
```

## 11. Модуль `dk_common.health`

### Назначение

Реализовать helper для единообразного `/health`.

На этом этапе healthcheck простой. Не нужно проверять PostgreSQL, Redis или MongoDB.

### Требуемый API

```python
from dk_common.health import build_health_response, create_health_router
```

### `build_health_response()`

Сигнатура:

```python
def build_health_response(
    service_name: str,
    environment: str = "local",
    version: str | None = None,
) -> dict:
    ...
```

Ожидаемый ответ:

```json
{
  "status": "ok",
  "service": "kitchen-service",
  "environment": "local",
  "version": "0.1.0"
}
```

Если версия неизвестна:

```json
{
  "status": "ok",
  "service": "kitchen-service",
  "environment": "local",
  "version": null
}
```

### `create_health_router()`

Сигнатура:

```python
def create_health_router(
    service_name: str,
    environment: str = "local",
    version: str | None = None,
):
    ...
```

Функция должна возвращать FastAPI `APIRouter` с endpoint:

```http
GET /health
```

### Пример подключения

```python
from fastapi import FastAPI
from dk_common.health import create_health_router

app = FastAPI()
app.include_router(
    create_health_router(
        service_name="kitchen-service",
        environment="local",
        version="0.1.0",
    )
)
```

### Критерии приемки

```text
- GET /health возвращает HTTP 200.
- Ответ содержит status=ok.
- Ответ содержит service.
- Ответ содержит environment.
- Ответ содержит version.
```

## 12. Модуль `dk_common.errors`

### Назначение

Реализовать базовую структуру инфраструктурных ошибок.

### Требуемый API

```python
from dk_common.errors import (
    DKCommonError,
    ConfigurationError,
    ExternalServiceError,
    HealthCheckError,
)
```

### Базовая ошибка

```python
class DKCommonError(Exception):
    code: str
    message: str
    details: dict | None
```

### Требования

Ошибки должны:

```text
1. Иметь человекочитаемое сообщение.
2. Иметь машинный code.
3. Опционально иметь details.
4. Нормально сериализоваться в dict через метод to_dict().
```

### Пример

```python
error = ConfigurationError(
    message="Invalid LOG_FORMAT",
    details={"allowed": ["json", "readable"]},
)

assert error.to_dict() == {
    "error": "configuration_error",
    "message": "Invalid LOG_FORMAT",
    "details": {"allowed": ["json", "readable"]},
}
```

### Критерии приемки

```text
- Все базовые ошибки создаются без падения.
- to_dict() возвращает стабильную структуру.
- Ошибки не содержат доменной логики.
```

## 16. Минимальный пример использования

После реализации должен работать такой FastAPI-пример:

```python
from fastapi import FastAPI

from dk_common.correlation import CorrelationIdMiddleware
from dk_common.health import create_health_router
from dk_common.logging import configure_logging

configure_logging(
    service_name="example-service",
    environment="local",
    log_level="INFO",
    log_format="readable",
)

app = FastAPI()
app.add_middleware(CorrelationIdMiddleware)
app.include_router(
    create_health_router(
        service_name="example-service",
        environment="local",
        version="0.1.0",
    )
)
```

`GET /health` должен вернуть:

```json
{
  "status": "ok",
  "service": "example-service",
  "environment": "local",
  "version": "0.1.0"
}
```

HTTP response должен содержать headers:

```http
X-Correlation-ID
X-Request-ID
```

Логи должны содержать минимум:

```text
timestamp
level
service
environment
message
correlation_id
request_id
```
