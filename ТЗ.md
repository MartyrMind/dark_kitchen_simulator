# Техническое задание

> Версия: KDS + Go Kitchen Scheduler Worker.
>
> Ключевые изменения:
>
> - Cooking Worker Service переосмыслен как **Kitchen Scheduler Worker на Go**.
> - Worker не готовит задачи и не ставит `in_progress` / `done`.
> - Worker читает Redis Streams, выбирает **конкретную station_id** и доставляет задачу в KDS.
> - KDS реализуется внутри **Kitchen Service**.
> - Работники станции или station-simulator видят задачи в KDS, берут их в работу через `claim` и завершают через `complete`.
> - **Fulfillment Service остается единственным владельцем глобальных бизнес-статусов заказов и kitchen tasks**.
> - Проект разрабатывается как **polyglot monorepo**: один репозиторий, но каждый deployable-сервис является отдельным проектом со своими зависимостями.
> - Для локальной Python-разработки используется **Poetry**, для сборки контейнеров — **uv**.
> - Общие технические компоненты для Python-сервисов выносятся в `libs/python/dk-common`.
> - Для логирования Python-сервисов используется **Loguru**; доменные события и важные audit/error-события сохраняются в MongoDB, а метрики отображаются в Grafana через Prometheus.

## Система управления выполнением заказов для dark kitchen

# 1. Назначение системы

Разработать микросервисную backend-систему для управления выполнением заказов в dark kitchen.

Система должна принимать заказы, раскладывать их на кухонные задачи, публиковать задачи в Redis Streams, диспетчеризовать их на конкретные станции через Go-based worker, отображать задачи на KDS-экранах станций, имитировать действия работников станции и отслеживать выполнение заказа до состояния `ready_for_pickup`.

Фактическое выполнение задачи не должно происходить через `sleep(time_to_cook)` внутри worker-а. Вместо этого задача должна проходить через KDS-flow:

```text
queued → displayed → in_progress → done
```

Где:

```text
queued       — задача создана Fulfillment Service и опубликована в Redis Stream;
displayed    — задача доставлена Go-worker-ом в KDS конкретной станции;
in_progress  — работник станции успешно взял задачу в работу через KDS claim;
done         — работник станции завершил задачу через KDS complete.
```

Система разрабатывается как учебный проект в рамках практического цикла по проектированию, разработке и эксплуатации микросервисной информационной системы. В проекте должны быть реализованы микросервисы, Docker-контейнеризация, Kubernetes-деплой, мониторинг, БД, асинхронная обработка и возможность Service Mesh.

# 2. Краткое описание бизнес-процесса

Dark kitchen принимает заказ от оператора или внешнего API-клиента. Заказ содержит список блюд и привязан к конкретной кухне.

После создания заказа система:

```text
1. Проверяет, существует ли кухня.
2. Проверяет, активна ли кухня.
3. Проверяет доступность блюд на этой кухне.
4. Получает рецепты блюд.
5. Разбивает заказ на kitchen tasks.
6. Сохраняет order и kitchen_tasks в Fulfillment Service.
7. Публикует задачи в Redis Streams по kitchen_id и station_type.
8. Kitchen Scheduler Worker на Go читает Redis Streams.
9. Worker проверяет актуальность задачи через Fulfillment Service API.
10. Worker проверяет готовность задачи к dispatch с учетом зависимостей recipe steps.
11. Worker запрашивает у Kitchen Service список dispatch candidates.
12. Worker выбирает конкретную station_id.
13. Worker доставляет задачу в KDS конкретной станции.
14. Fulfillment Service фиксирует task.status = displayed.
15. Работник станции видит задачу в KDS.
16. Работник или station-simulator вызывает claim.
17. Kitchen Service / KDS атомарно проверяет capacity и закрепляет задачу за station_worker_id.
18. Kitchen Service вызывает Fulfillment Service для перехода displayed → in_progress.
19. Работник или station-simulator имитирует приготовление.
20. Работник или station-simulator вызывает complete.
21. Kitchen Service вызывает Fulfillment Service для перехода in_progress → done.
22. Kitchen Service освобождает busy slot станции.
23. Fulfillment Service проверяет, все ли задачи заказа завершены.
24. Если все задачи завершены, order.status = ready_for_pickup.
25. Оператор или API-клиент отмечает передачу заказа курьеру.
```

# 3. Цели MVP

MVP должен позволять:

```text
1. Создать кухни и станции.
2. Создать меню и рецепты блюд.

6. Запустить Kitchen Scheduler Worker на Go.
7. Диспетчеризовать задачи на конкретные станции.
8. Отображать задачи в KDS внутри Kitchen Service.
9. Имитировать работников станции через station-simulator-service.
10. Брать задачи в работу через KDS claim.
11. Завершать задачи через KDS complete.
12. Не допускать превышения station capacity.
13. Не допускать двойного claim одной задачи.
14. Хранить текущее состояние в PostgreSQL.
15. Хранить историю событий в MongoDB.
16. Экспортировать метрики для Prometheus.
17. Запускать систему через docker-compose.
18. Развернуть систему в Kubernetes / Minikube.
```

# 4. Архитектура системы

## 4.1. Deployable-сервисы

Система состоит из core-сервисов и demo-сервиса:

| Сервис | Язык / стек | Назначение |
|---|---|---|
| **Kitchen Service** | Python / FastAPI | Управление кухнями, станциями, capacity и KDS API |
| **Menu Service** | Python / FastAPI | Управление меню, блюдами, рецептами и доступностью блюд по кухням |
| **Fulfillment Service** | Python / FastAPI | Прием заказов, создание kitchen tasks, управление глобальными бизнес-статусами заказов и задач |
| **Kitchen Scheduler Worker** | Go | Чтение Redis Streams, выбор конкретной станции, dispatch задач в KDS, retry/backoff/DLQ |
| **Station Simulator Service** | Python или Go | Demo-сервис, имитирующий работников станции, которые читают KDS, claim-ят задачи и завершают их |

## 4.2. Архитектурные роли

### Fulfillment Service

Fulfillment Service владеет:

```text
orders
order_items
kitchen_tasks
business task statuses
order statuses
SLA deadlines
переходом заказа в ready_for_pickup
```

Fulfillment Service является **единственным сервисом**, который изменяет глобальные бизнес-статусы kitchen tasks:

```text
created
queued
displayed
in_progress
done
failed
retrying
cancelled
```

Ни Kitchen Service, ни Go-worker не должны напрямую писать в таблицу `kitchen_tasks`.

### Kitchen Service

Kitchen Service владеет:

```text
kitchens
stations
station capacity
busy slots
KDS
локальными station_tasks
claim / complete действиями работников станции
```

Kitchen Service реализует KDS API. KDS отвечает за то, что видит сотрудник станции:

```text
- список задач станции;
- claim задачи;
- complete задачи;
- fail задачи;
- локальный статус station_task;
- контроль busy_slots < capacity.
```

### Kitchen Scheduler Worker на Go

Kitchen Scheduler Worker не является поваром и не выполняет задачу. Его роль — асинхронная диспетчеризация:

```text
Redis Streams → выбор station_id → KDS delivery → mark displayed
```

Worker отвечает за:

```text
1. Чтение Redis Streams через consumer group.
2. Проверку актуальности задачи через Fulfillment API.
3. Проверку dispatch-readiness с учетом dependencies.
4. Выбор конкретной station_id на основе состояния Kitchen/KDS.
5. Контроль visible backlog станции.
6. Доставку задачи в KDS конкретной станции.
7. Идемпотентность dispatch.
8. Retry / backoff / DLQ.
9. Метрики dispatch-процесса.
```

Worker не должен:

```text
- напрямую читать или писать PostgreSQL Fulfillment Service;
- ставить task.status = in_progress;
- ставить task.status = done;
- выполнять sleep(time_to_cook);
- освобождать busy slot станции;
- переводить order в ready_for_pickup.
```

### Station Simulator Service

Station Simulator Service имитирует реальных работников станции. Он не содержит бизнес-логики заказа и не ходит напрямую в Fulfillment Service.

Он работает только с KDS API Kitchen Service:

```text
GET /kds/stations/{station_id}/tasks
POST /kds/stations/{station_id}/tasks/{task_id}/claim
POST /kds/stations/{station_id}/tasks/{task_id}/complete
```

# 5. Технологический стек

## 5.1. API-сервисы

```text
Python 3.11+
FastAPI

Pydantic
SQLAlchemy
Alembic
```

## 5.2. Kitchen Scheduler Worker

```text
Go 1.22+
go-redis / rueidis
net/http или resty для HTTP-клиентов
golang.org/x/sync для worker pools
prometheus/client_golang
slog / zerolog для structured logging
```

Go выбран для worker-а, потому что сервис является long-running high-concurrency компонентом, хорошо ложится на goroutines, worker pools, Redis Streams consumer group, retry loops и Kubernetes scaling.

## 5.3. Хранилища и инфраструктура

| Компонент | Назначение |
|---|---|
| **PostgreSQL** | Транзакционное состояние сервисов |
| **MongoDB** | Event log: order_events, task_events, station_events, kds_events |
| **Redis Streams** | Очереди kitchen tasks |
| **Redis** | Вспомогательные counters/cache, если нужно |
| **Docker / docker-compose** | Локальный запуск |
| **Kubernetes / Minikube** | Деплой |
| **Prometheus + Grafana** | Метрики и визуализация |
| **Linkerd / Istio** | Опциональный Service Mesh |

Важно: даже если в MVP используется один физический PostgreSQL, логически каждый сервис владеет своими таблицами или схемой. Сервисы не читают и не пишут чужие таблицы напрямую.

## 5.4. Управление зависимостями в монорепозитории

Проект должен разрабатываться как **polyglot monorepo**:

```text
один Git-репозиторий
  несколько независимых deployable-сервисов
  общая инфраструктура для локального запуска и Kubernetes
  общие технические библиотеки
  общие API/событийные контракты
```

При этом монорепозиторий не должен превращаться в единый Python-проект. Каждый сервис является отдельной единицей поставки и имеет собственные зависимости:

```text
services/kitchen-service/pyproject.toml
services/menu-service/pyproject.toml
services/fulfillment-service/pyproject.toml
services/station-simulator-service/pyproject.toml
services/kitchen-scheduler-worker/go.mod
```

Для Python-сервисов применяется следующий подход:

```text
Локальная разработка:
  Poetry
  pyproject.toml
  poetry.lock
  poetry install
  poetry run pytest
  poetry run uvicorn ...

Docker / Kubernetes image build:
  uv
  uv pip install
  быстрая установка зависимостей в контейнере
```

`pyproject.toml` является основным источником описания зависимостей Python-сервиса. `poetry.lock` фиксирует зависимости для локальной разработки. Для контейнеров допускается установка через `uv pip install` из `pyproject.toml`. Если требуется полностью воспроизводимая контейнерная сборка, сервис должен дополнительно генерировать `requirements.lock.txt` из Poetry lock-файла и устанавливать его через `uv pip install -r requirements.lock.txt`.

Пример локальной разработки:

```bash
cd services/kitchen-service
poetry install
poetry run uvicorn app.main:app --reload
poetry run pytest
```

Пример контейнерной установки:

```dockerfile
RUN pip install uv
RUN uv pip install --system /repo/libs/python/dk-common
RUN uv pip install --system /repo/services/kitchen-service
```

## 5.5. Общая Python-библиотека `libs/python/dk-common`

В репозитории должна быть общая техническая библиотека:

```text
libs/
  python/
    dk-common/
      pyproject.toml
      dk_common/
        __init__.py
        logging.py
        settings.py
        metrics.py
        http.py
        mongo_events.py
```

Эта библиотека предназначена только для технического кода, который одинаково нужен нескольким Python-сервисам.

В `dk-common` разрешено размещать:

```text
- настройку Loguru;
- JSON-форматирование логов;
- correlation_id / request_id middleware;
- базовые настройки Pydantic Settings;
- общие helpers для Prometheus metrics;
- общую обертку над httpx client;
- общую обертку для записи доменных событий в MongoDB;
- healthcheck helpers;
- общие исключения инфраструктурного уровня.
```

В `dk-common` запрещено размещать:

```text
- SQLAlchemy-модели конкретных сервисов;
- Alembic-миграции;
- бизнес-логику заказов, рецептов, станций или KDS;
- репозитории сервисов;
- use cases сервисов;
- код, который делает сервисы зависимыми от чужой внутренней модели данных.
```

Правило:

```text
Общая библиотека может знать, как логировать, измерять и отправлять HTTP-запросы.
Общая библиотека не должна знать, что такое order, kitchen_task, station, recipe или KDS.
```

Пример подключения `dk-common` в Python-сервисе через Poetry:

```toml
[tool.poetry.dependencies]
python = "^3.11"
dk-common = { path = "../../libs/python/dk-common", develop = true }
fastapi = "^0.115.0"
uvicorn = { extras = ["standard"], version = "^0.30.0" }
```

Пример импорта:

```python
from dk_common.logging import configure_logging
from dk_common.metrics import setup_metrics
```

Запрещено подключать общую библиотеку через `sys.path.append` или относительные импорты вида `../../../libs/...`.

## 5.6. Контракты между сервисами

API-контракты и схемы сообщений должны храниться отдельно от реализации сервисов:

```text
contracts/
  openapi/
    kitchen-service.yaml
    menu-service.yaml
    fulfillment-service.yaml

  schemas/
    redis-task-message.schema.json
    order-events.schema.json
    task-events.schema.json
    kds-events.schema.json
    station-events.schema.json
```

Назначение `contracts/`:

```text
1. Зафиксировать HTTP-контракты между сервисами.
2. Зафиксировать формат сообщений Redis Streams.
3. Зафиксировать формат событий MongoDB event log.
4. Упростить генерацию клиентов или тестовых фикстур.
5. Не смешивать публичные контракты с внутренними моделями сервисов.
```

| Поле | Тип | Описание |

# 8. Redis Streams

## 8.1. Очереди

Fulfillment Service публикует задачи в Redis Streams по `kitchen_id` и `station_type`.

Формат stream:

```text
stream:kitchen:{kitchen_id}:station:{station_type}
```

Примеры:

```text
stream:kitchen:msk_01:station:grill
stream:kitchen:msk_01:station:fryer
stream:kitchen:msk_01:station:packaging
```

## 8.2. Сообщение в Redis Stream

```json
{
  "task_id": "task_456",
  "order_id": "ord_123",
  "kitchen_id": "msk_01",
  "station_type": "grill",
  "operation": "cook_patty",
  "menu_item_id": "burger",
  "menu_item_name": "Burger",
  "estimated_duration_seconds": 480,
  "pickup_deadline": "2026-04-27T18:45:00Z",
  "attempt": 1
}
```

Redis-сообщение является транспортной подсказкой. Источник истины по статусу задачи — Fulfillment Service.

## 8.3. Consumer group

Kitchen Scheduler Worker использует consumer group:

```text
group:kitchen-scheduler-workers
```

Каждая реплика worker-а является consumer-ом:

```text
scheduler-worker-1
scheduler-worker-2
scheduler-worker-3
```

## 8.4. ACK policy

Worker должен делать `XACK` только после того, как:

```text
1. Задача успешно доставлена в KDS конкретной станции.
2. Fulfillment Service успешно перевел задачу queued → displayed.
```

Если KDS delivery успешен, но `mark-displayed` не прошел, worker должен повторить `mark-displayed` идемпотентно.

Если `mark-displayed` говорит, что задача уже `displayed`, worker может ACK-нуть Redis message.

## 8.5. Retry и DLQ

Если dispatch не удался:

```text
1. Worker увеличивает dispatch attempt.
2. Делает delayed retry с backoff.
3. После MAX_DISPATCH_ATTEMPTS отправляет сообщение в DLQ.
4. Вызывает Fulfillment Service: dispatch-failed.
```

DLQ stream:

```text
stream:kitchen:{kitchen_id}:station:{station_type}:dlq
```

# 10. Kitchen Scheduler Worker на Go

## 10.1. Назначение

Kitchen Scheduler Worker — отдельный Go-сервис, который читает kitchen tasks из Redis Streams и диспетчеризует их на конкретные станции через KDS API.

Он не является владельцем задачи и не меняет PostgreSQL напрямую.

## 10.2. Основные функции

```text
1. Читать Redis Streams через XREADGROUP.
2. Поддерживать несколько реплик в Kubernetes.
3. Проверять task snapshot через Fulfillment Service.
4. Пропускать задачи, которые уже не находятся в dispatchable status.
5. Проверять dispatch-readiness и recipe dependencies.
6. Получать dispatch candidates из Kitchen Service / KDS.
7. Выбирать конкретную station_id.
8. Контролировать visible backlog.
9. Доставлять задачу в KDS конкретной станции.
10. Идемпотентно обрабатывать повторную доставку.
11. Вызывать Fulfillment mark-displayed.
12. ACK Redis message только после успешной фиксации displayed.
13. Выполнять retry/backoff/DLQ при ошибках.
14. Экспортировать dispatch metrics.
```

## 10.3. Dispatch algorithm

```text
1. Worker читает сообщение из Redis Stream через XREADGROUP.
2. Из сообщения получает task_id, kitchen_id, station_type, order_id, pickup_deadline.
3. Worker вызывает Fulfillment Service: GET /internal/tasks/{task_id}.
4. Если task.status ∈ {displayed, in_progress, done, cancelled, failed}:
   - XACK;
   - skip.
5. Если task.status не dispatchable:
   - retry или skip согласно правилам.
6. Worker вызывает Fulfillment Service: GET /internal/tasks/{task_id}/dispatch-readiness.
7. Если ready_to_dispatch = false:
   - delayed retry;
   - XACK старого сообщения;
   - skip.
8. Worker вызывает Kitchen Service:
   GET /internal/kds/dispatch-candidates?kitchen_id=...&station_type=...
9. Worker фильтрует кандидатов:
   - status = available;
   - health = ok;
   - visible_backlog_size < visible_backlog_limit.
10. Если кандидатов нет:
   - delayed retry;
   - записать DispatchRetried;
   - XACK старого сообщения.
11. Worker выбирает конкретную station_id.
12. Worker вызывает:
   POST /internal/kds/stations/{station_id}/tasks
   с idempotency_key = task_id:dispatch:v1.
13. Если KDS вернул success или already exists:
   - worker вызывает Fulfillment:
     POST /internal/tasks/{task_id}/mark-displayed.
14. Если mark-displayed успешен или задача уже displayed:
   - XACK Redis message.
15. Если KDS или Fulfillment временно недоступны:
   - retry/backoff.
16. Если достигнут MAX_DISPATCH_ATTEMPTS:
   - отправить в DLQ;
   - вызвать Fulfillment dispatch-failed.
```

## 10.4. Выбор станции

Для MVP:

```text
1. Взять все станции-кандидаты нужного station_type.
2. Отфильтровать unavailable / maintenance.
3. Отфильтровать станции, где visible_backlog_size >= visible_backlog_limit.
4. Выбрать станцию с минимальным visible_backlog_size.
5. При равенстве выбрать станцию с минимальным busy_slots.
```

Расширенный score:

```text
score = visible_backlog_size * 10 + busy_slots * 20 + average_delay_seconds * 0.1
```

Выбирается станция с минимальным score.

# 11. Station Simulator Service

## 11.1. Назначение

Station Simulator Service имитирует работников станции. Он создает несколько виртуальных workers, каждый из которых закреплен за конкретной станцией.

Пример виртуальных работников:

```text
grill-worker-1 → station_id=grill_1
grill-worker-2 → station_id=grill_1
fryer-worker-1 → station_id=fryer_1
packaging-worker-1 → station_id=packaging_1
```

## 11.2. Алгоритм симуляции

```text
1. Запустить N виртуальных работников.
2. Каждый worker в цикле вызывает GET /kds/stations/{station_id}/tasks.
3. Выбирает задачу со status = displayed.
4. Вызывает POST /kds/stations/{station_id}/tasks/{task_id}/claim.
5. Если получил 200 OK:
   - задача считается взятой в работу;
   - Fulfillment уже получил start;
   - task.status = in_progress.
6. Симулятор ждет simulated duration:
   actual_duration = estimated_duration_seconds * random(0.7, 1.4) / SPEED_FACTOR.
7. Вызывает POST /kds/stations/{station_id}/tasks/{task_id}/complete.
8. Переходит к следующей задаче.
```

В отличие от Go-worker-а, station-simulator может использовать `sleep`, потому что он имитирует действия человека, а не является системой бизнес-статусов.

# 12. Конкурентность и consistency

## 12.1. Двойной claim одной задачи

Даже если Go-worker назначает задачу на конкретную station_id, несколько работников одной станции могут одновременно нажать claim.

KDS должен защищать claim атомарно.

Пример SQL-подхода:

```sql
BEGIN;

SELECT *
FROM stations
WHERE id = :station_id
FOR UPDATE;

-- Проверить busy_slots < capacity

UPDATE kds_station_tasks
SET status = 'claimed',
    claimed_by = :station_worker_id,
    claimed_at = now(),
    updated_at = now()
WHERE task_id = :task_id
  AND station_id = :station_id
  AND status = 'displayed';

-- Если affected rows = 0, задача уже взята или недоступна.

UPDATE stations
SET busy_slots = busy_slots + 1,
    updated_at = now()
WHERE id = :station_id;

COMMIT;
```

Если affected rows = 0:

```http
409 Conflict
```

```json
{
  "error": "task_already_claimed"
}
```

## 12.2. Capacity

Capacity учитывается на этапе `claim`, а не на этапе dispatch.

Это означает:

```text
Go-worker может показать задачу на станции,
но busy slot занимается только когда работник реально берет задачу в работу.
```

Чтобы KDS не переполнялся, worker контролирует `visible_backlog_limit`.

Пример:

```text
station.capacity = 2
visible_backlog_limit = 4
```

То есть на экране станции может быть максимум 4 задачи:

```text
2 потенциально в работе + 2 ожидающие
```

## 12.3. Ошибка Fulfillment /start после успешного claim

Если KDS успешно сделал claim и занял busy slot, но Fulfillment `/start` временно недоступен:

```text
1. KDS переводит station_task в claim_pending_start или оставляет claimed с pending flag.
2. KDS повторяет вызов /start с idempotency key.
3. Если /start окончательно не удался, KDS переводит station_task в failed и освобождает busy slot.
```

Для MVP допустимо синхронно вызвать `/start` и при ошибке откатить транзакцию claim, если реализация не усложняется.

## 12.4. Ошибка Fulfillment /complete

Если KDS получил complete от работника, но Fulfillment `/complete` временно недоступен:

```text
1. KDS сохраняет station_task.status = completion_pending.
2. Повторяет /complete с idempotency key.
3. Освобождает busy slot после успешного complete.
```

Для MVP допустимо:

```text
- вызвать Fulfillment /complete;
- если успех — station_task.completed и release busy slot;
- если ошибка — вернуть 503 и оставить задачу claimed.
```

# 14. Нефункциональные требования

## 14.1. Масштабируемость

Система должна поддерживать:

```text
1. Несколько кухонь.
2. Несколько станций одного типа на кухне.
3. Разный capacity у разных станций.
4. Несколько реплик Kitchen Scheduler Worker.
5. Несколько виртуальных работников станции.
6. Разделение Redis Streams по kitchen_id и station_type.
```

## 14.2. Отказоустойчивость

Система должна:

```text
1. Не создавать дубликаты KDS-задач при повторной доставке.
2. Не позволять двум работникам claim-ить одну задачу.
3. Не превышать station capacity.
4. Поддерживать retry/backoff для dispatch.
5. Поддерживать DLQ для задач, которые не удалось доставить в KDS.
6. Не терять задачу при падении worker-а.
7. Сохранять event log для диагностики.
```

## 14.3. Идемпотентность

Идемпотентными должны быть:

```text
POST /internal/kds/stations/{station_id}/tasks
POST /internal/tasks/{task_id}/mark-displayed
POST /internal/tasks/{task_id}/start
POST /internal/tasks/{task_id}/complete
```

## 14.4. Наблюдаемость

Каждый deployable-сервис должен иметь:

```http
GET /health
GET /metrics
```

Наблюдаемость системы строится на трех уровнях:

```text
1. Metrics — числовые метрики Prometheus, отображаются в Grafana.
2. Structured logs — структурированные технические логи в stdout, пригодные для Docker/Kubernetes.
3. Event log — доменные и audit-события в MongoDB для диагностики бизнес-процессов.
```

Важно разделять:

```text
MongoDB event log:
  - история бизнес-событий и audit-событий;
  - используется для проверки цепочки выполнения заказа;
  - хранит OrderCreated, TaskQueued, KdsTaskClaimed, TaskCompleted и т.д.;
  - может хранить важные error/audit-события, которые имеют бизнес-смысл.

Application logs:
  - технические логи работы приложения;
  - пишутся в stdout/stderr контейнера;
  - содержат debug/info/warning/error сообщения;
  - используются для диагностики кода, HTTP-запросов, исключений, retry и внутренних ошибок.

Grafana:
  - в обязательной части MVP показывает метрики Prometheus;
  - для просмотра application logs через Grafana дополнительно нужен Loki или другой log backend;
  - без Loki логи смотрятся через docker logs / kubectl logs, а MongoDB используется для event/audit истории.
```

MongoDB не должна использоваться как замена технической системе логирования. Не нужно писать в MongoDB каждый `info` или `debug` лог. MongoDB используется как **event store / audit trail** для важных событий домена и критичных диагностических событий.

Для Python-сервисов техническое логирование реализуется через `Loguru` и общую настройку из `libs/python/dk-common`.

Для Go worker-а допускается использовать `slog` или `zerolog`, но формат логов должен быть совместимым по ключевым полям с Python-сервисами.

Обязательные поля структурированного лога:

```text
timestamp
level
service
environment
message
correlation_id
request_id
order_id nullable
task_id nullable
kitchen_id nullable
station_id nullable
worker_id nullable
error nullable
```

Технические логи должны писаться в stdout/stderr контейнера. Это основной путь логирования для Docker и Kubernetes.

В MongoDB не нужно писать каждый debug/info лог. В MongoDB должны попадать:

```text
1. Доменные события: OrderCreated, TaskQueued, TaskDisplayed, TaskStarted, TaskCompleted и т.д.
2. KDS-события: KdsTaskDisplayed, KdsTaskClaimed, KdsTaskCompleted, KdsTaskClaimRejected.
3. Station events: StationCapacityChanged, StationBusySlotOccupied, StationBusySlotReleased.
4. Важные audit/error-события: DispatchFailed, DispatchRetried, ExternalServiceUnavailable, FulfillmentCallbackFailed.
```

Grafana в MVP используется прежде всего для визуализации Prometheus-метрик:

```text
- dispatch latency;
- Redis pending messages;
- DLQ messages;
- KDS visible backlog;
- station utilization;
- task duration;
- task delay;
- order ready count;
- HTTP request rate/error rate/latency.
```

Для просмотра логов в Grafana допускается опционально добавить Loki. В MVP Loki не является обязательным, если технические логи доступны через Docker/Kubernetes logs, а бизнес-события доступны через MongoDB.

## 14.5. Correlation ID и трассировка бизнес-процесса

Каждый внешний HTTP-запрос должен получать `correlation_id`.

`correlation_id` должен передаваться дальше во все внутренние HTTP-вызовы:

```text
Fulfillment Service → Menu Service
Fulfillment Service → Kitchen Service
Kitchen Scheduler Worker → Fulfillment Service
Kitchen Scheduler Worker → Kitchen Service / KDS
Kitchen Service / KDS → Fulfillment Service
Station Simulator → Kitchen Service / KDS
```

`correlation_id` должен попадать:

```text
- в structured logs;
- в MongoDB events;
- в HTTP response headers;
- по возможности в Prometheus exemplars или labels только там, где это не создает high-cardinality проблему.
```

В Prometheus labels запрещено добавлять high-cardinality значения:

```text
нельзя:
  order_id
  task_id
  request_id
  correlation_id

можно:
  service
  method
  path_template
  status
  kitchen_id для бизнес-метрик MVP
  station_type
  station_id для station-level метрик MVP
```

## 18.1. Уровни тестирования

Тестирование должно быть разделено на несколько уровней.

```text
Unit tests:
  - тестируют чистую бизнес-логику внутри одного сервиса;
  - не поднимают реальные PostgreSQL, Redis, MongoDB;
  - используют mocks/fakes для внешних клиентов.

Component tests:
  - тестируют один сервис целиком через его HTTP API;
  - поднимают реальные зависимости этого сервиса: PostgreSQL, Redis или MongoDB;
  - внешние сервисы заменяются stub/mock HTTP-серверами.

Contract tests:
  - проверяют, что сервисы соблюдают OpenAPI/JSON schemas из contracts/;
  - отдельно проверяют формат Redis Stream messages;
  - отдельно проверяют формат MongoDB events.

Integration tests:
  - поднимают несколько настоящих сервисов вместе;
  - проверяют межсервисные взаимодействия через HTTP, Redis Streams, MongoDB;
  - запускаются медленнее unit/component тестов.

End-to-end / business-flow tests:
  - поднимают всю систему через docker-compose;
  - проверяют полный бизнес-сценарий от создания заказа до ready_for_pickup.
```

## 18.3. Как писать integration tests между сервисами

Для integration tests допускаются два подхода.

### Подход A — docker-compose integration tests

Этот подход рекомендуется для MVP и защиты.

```text
1. Собрать docker images сервисов.
2. Поднять docker-compose.integration.yml.
3. Дождаться готовности /health всех сервисов.
4. Прогнать миграции.
5. Засеять тестовые кухни, станции, меню и рецепты.
6. Выполнить HTTP-запросы к public API.
7. Проверить состояния через public/internal API, Redis, MongoDB и метрики.
8. Остановить compose и удалить volumes.
```

Пример структуры:

```text
deploy/compose/docker-compose.integration.yml
tests/integration/conftest.py
tests/integration/test_order_to_kds_flow.py
```

Плюсы:

```text
- максимально похоже на реальный запуск;
- удобно демонстрировать;
- хорошо проверяет docker-compose и env-конфигурацию.
```

Минусы:

```text
- медленнее unit/component тестов;
- сложнее отлаживать;
- требует аккуратной очистки данных между тестами.
```

### Подход B — Testcontainers

Альтернативно можно использовать Testcontainers.

```text
1. Pytest поднимает PostgreSQL, Redis, MongoDB через testcontainers.
2. Сервисы запускаются как процессы или контейнеры.
3. Тесты обращаются к сервисам через HTTP.
```

Плюсы:

```text
- удобно для CI;
- можно точнее контролировать жизненный цикл зависимостей.
```

Минусы:

```text
- сложнее настроить для polyglot-проекта с Python и Go;
- для учебного MVP docker-compose обычно проще.
```

Для данного проекта основной рекомендуемый путь — **docker-compose integration tests**, а Testcontainers можно добавить позже.

## 18.4. Сквозной тест полного бизнес-флоу

Главный end-to-end тест должен проверять цепочку:

```text
create kitchen
→ create stations
→ create menu items
→ create recipe steps
→ create order
→ Fulfillment creates kitchen_tasks
→ tasks are published to Redis Streams
→ Go worker dispatches tasks to KDS
→ KDS shows tasks on stations
→ station simulator or test client claims tasks
→ Fulfillment receives /start
→ station simulator or test client completes tasks
→ Fulfillment receives /complete
→ order.status becomes ready_for_pickup
→ MongoDB contains expected events
→ Prometheus metrics are exposed
```

Пример псевдокода:

```python
def test_order_to_ready_for_pickup(api, eventually):
    kitchen = api.kitchen.create_kitchen(name="msk_01")
    grill = api.kitchen.create_station(kitchen.id, type="grill", capacity=1)
    packaging = api.kitchen.create_station(kitchen.id, type="packaging", capacity=1)

    burger = api.menu.create_item("burger")
    api.menu.add_recipe_step(burger.id, station_type="grill", operation="cook_patty", order=1)
    api.menu.add_recipe_step(burger.id, station_type="packaging", operation="pack_burger", order=2)
    api.menu.set_available(kitchen.id, burger.id, True)

    order = api.fulfillment.create_order(kitchen.id, items=[{"menu_item_id": burger.id, "quantity": 1}])

    eventually(lambda: api.fulfillment.order_has_tasks(order.id, expected_count=2))
    eventually(lambda: api.kds.station_has_displayed_task(grill.id))

    task = api.kds.first_displayed_task(grill.id)
    api.kds.claim(grill.id, task.task_id, worker_id="test-grill-worker")
    api.kds.complete(grill.id, task.task_id, worker_id="test-grill-worker")

    eventually(lambda: api.kds.station_has_displayed_task(packaging.id))

    task = api.kds.first_displayed_task(packaging.id)
    api.kds.claim(packaging.id, task.task_id, worker_id="test-packaging-worker")
    api.kds.complete(packaging.id, task.task_id, worker_id="test-packaging-worker")

    eventually(lambda: api.fulfillment.get_order(order.id).status == "ready_for_pickup")
```

Так как система асинхронная, тесты не должны использовать жесткие `sleep(10)`. Нужно использовать helper `eventually()`:

```text
eventually:
  - повторяет проверку до timeout;
  - имеет небольшой polling interval;
  - при timeout показывает последнее состояние системы.
```

## 18.5. Тестирование Redis Streams и worker-а

Для worker-а нужны отдельные тесты:

```text
1. Worker читает сообщение из Redis Stream.
2. Worker проверяет task snapshot в Fulfillment.
3. Worker не dispatch-ит задачу, если task.status уже displayed/done/cancelled.
4. Worker не dispatch-ит packaging до завершения grill.
5. Worker выбирает station_id с минимальным visible_backlog_size.
6. Worker делает XACK только после успешного KDS delivery и mark-displayed.
7. Worker отправляет задачу в DLQ после MAX_DISPATCH_ATTEMPTS.
```

Для unit tests Go worker-а HTTP-клиенты Fulfillment/Kitchen должны быть интерфейсами, чтобы их можно было заменить fake-реализациями.

Для integration tests worker должен запускаться настоящим контейнером и работать с настоящими Redis, Kitchen Service и Fulfillment Service.

## 18.6. Тестирование конкурентности

Обязательные concurrency tests:

```text
Double claim:
  - одна KDS-задача displayed;
  - два параллельных запроса claim;
  - только один получает 200;
  - второй получает 409 task_already_claimed;
  - busy_slots увеличивается только на 1.

Capacity:
  - station.capacity = 1;
  - две KDS-задачи displayed;
  - два параллельных claim на разные задачи;
  - только одна задача переходит в claimed;
  - busy_slots не превышает 1.

Idempotent dispatch:
  - два одинаковых POST /internal/kds/stations/{station_id}/tasks с одним idempotency_key;
  - создается только одна kds_station_task;
  - оба ответа возвращают один и тот же kds_task_id.
```

Эти тесты должны быть либо component tests Kitchen Service с реальным PostgreSQL, либо integration tests с поднятой системой.

## 18.7. Тестирование MongoDB events и логов

Для MongoDB нужно тестировать не технические логи, а бизнес/audit-события.

Проверять нужно:

```text
1. При создании заказа появляется OrderCreated.
2. При публикации задачи появляется TaskQueued.
3. При dispatch появляется TaskDisplayed и KdsTaskDisplayed.
4. При claim появляется KdsTaskClaimed и TaskStarted.
5. При complete появляется KdsTaskCompleted и TaskCompleted.
6. После завершения всех задач появляется OrderReadyForPickup.
7. У событий есть correlation_id, service, event_type, created_at и payload.
```

Application logs через Loguru не нужно проверять как бизнес-состояние. Достаточно unit/component теста, что `configure_logging()` добавляет обязательные поля и пишет JSON в контейнерном режиме.

## 18.8. Тестирование метрик

Метрики проверяются на уровне smoke/integration tests:

```text
1. GET /metrics каждого Python-сервиса возвращает 200.
2. Go worker открывает PROMETHEUS_PORT и отдает метрики.
3. После создания заказа растет orders_created_total.
4. После dispatch растет dispatch_success_total.
5. После claim растет kds_claim_success_total.
6. После complete растет tasks_completed_total.
```

В тестах не нужно проверять все значения метрик до каждой цифры. Достаточно проверить наличие метрик и изменение ключевых counters.

## 18.9. Минимальный набор тестов для MVP

Минимально к защите должны быть реализованы:

```text
Unit/component:
  1. Menu Service: recipe_steps возвращаются в правильном порядке.
  2. Fulfillment Service: order раскладывается на kitchen_tasks и task_dependencies.
  3. Kitchen Service: idempotent KDS delivery не создает дубль.
  4. Kitchen Service: double claim невозможен.
  5. Kitchen Service: capacity не превышается.

Integration / E2E:
  6. Order → Redis → Worker → KDS → claim → complete → ready_for_pickup.
  7. Packaging не dispatch-ится до завершения grill.
  8. MongoDB содержит expected events по заказу.
  9. /metrics доступны у всех сервисов.
```

# 23. Критерии готовности

Проект считается готовым, если:

```text
1. Все сервисы запускаются через docker-compose.
2. Можно создать кухню, станции, меню и рецепт.
3. Можно создать заказ.
4. Заказ автоматически раскладывается на kitchen tasks.
5. Задачи попадают в Redis Streams.
6. Kitchen Scheduler Worker на Go читает Redis Streams.
7. Worker выбирает конкретную station_id.
8. Задачи появляются в KDS конкретных станций.
9. Fulfillment task.status переходит queued → displayed.
10. Station Simulator или ручной API вызов может claim-ить задачу.
11. Fulfillment task.status переходит displayed → in_progress.
12. Capacity станции не превышается.
13. Двойной claim одной задачи невозможен.
14. Station Simulator или ручной API вызов может complete-ить задачу.
15. Fulfillment task.status переходит in_progress → done.
16. После завершения всех задач order.status = ready_for_pickup.
17. События записываются в MongoDB.
18. Метрики доступны на /metrics.
19. Есть минимум 5 unit/component автотестов, включая KDS claim, capacity и idempotent dispatch.
20. Есть минимум 1 integration/E2E тест полного business-flow: order → Redis → Worker → KDS → claim → complete → ready_for_pickup.
21. Есть тест или проверяемый сценарий, что MongoDB содержит expected business/audit events.
22. Есть тест или smoke-check, что /metrics доступны у всех сервисов.
23. Есть Kubernetes-манифесты.
24. Есть Ingress для Fulfillment Service и KDS read endpoint.
25. Есть Grafana dashboard или скриншоты метрик.
```

# 24. Краткая формулировка темы для отчета

**Тема:**

**Система управления выполнением заказов для dark kitchen с Redis Streams, Go-based scheduler worker и KDS-моделью станций**

**Краткое описание:**

Система предназначена для управления процессом приготовления заказов в dark kitchen. Она принимает заказы, раскладывает их на kitchen tasks согласно рецептам, публикует задачи в Redis Streams, диспетчеризует их через Go-based Kitchen Scheduler Worker на конкретные станции и отображает в KDS API, которым владеет Kitchen Service. Работники станции или station-simulator берут задачи в работу через claim и завершают через complete. Fulfillment Service остается владельцем глобальных бизнес-статусов задач и заказов, контролирует SLA и переводит заказ в `ready_for_pickup` после завершения всех задач.
