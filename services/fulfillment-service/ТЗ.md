# Agent Task Spec: Stage 4 - Fulfillment Service without Redis dispatch

Encoding note:

## 0. Context

The project is a polyglot monorepo for a dark kitchen order fulfillment system.

Previous stages should already exist:

```text
Stage 1:
  libs/python/dk-common
  common settings, logging, correlation middleware, health helper, errors

Stage 2:
  services/kitchen-service
  kitchens and stations management without KDS

Stage 3:
  services/menu-service
  menu items, recipe steps, kitchen menu availability
```

This stage is about implementing:

```text
services/fulfillment-service
```

Fulfillment Service owns:

```text
orders
order_items
kitchen_tasks
task_dependencies
global task business statuses
order statuses
```

In the final system, Fulfillment Service is the only service that changes global kitchen task statuses.

In this stage, do not implement Redis dispatch yet.

---

## 1. Goal

Implement Fulfillment Service without Redis publishing and without dispatch.

At the end of this stage, the system must support:

```text
1. Create an order through POST /orders.
2. Validate kitchen through Kitchen Service.
3. Validate menu item availability through Menu Service.
4. Get recipe steps from Menu Service.
5. Create order record.
6. Create order_items.
7. Create kitchen_tasks from recipe steps.
8. Create task_dependencies based on recipe step order.
9. Return order details.
10. Return order tasks.
```

Important stage boundary:

```text
Kitchen tasks must be created with status = created.
Do not publish tasks to Redis in this stage.
Do not set task.status = queued in this stage by default.
Queued status and Redis Streams belong to Stage 5.
```

Optional feature flag:

```text
ENABLE_REDIS_PUBLISHING=false
```

If the flag exists, keep it false by default and do not implement publishing yet.
Do not add Redis as a required runtime dependency in this stage.

## 3. Out of scope

Do not implement in this stage:

```text
- Redis connection
- Redis Streams publishing
- XADD
- task.status = queued
- Kitchen Scheduler Worker
- KDS delivery
- KDS APIs
- station selection
- mark-displayed
- task start
- task complete
- ready_for_pickup transition
- MongoDB event log
- Prometheus business metrics
- DLQ
- retry/backoff
- Station Simulator
- Kubernetes manifests
```

Do not create KDS state in Fulfillment Service.

Do not write to Kitchen Service database.

Do not write to Menu Service database.

Fulfillment Service must communicate with Kitchen Service and Menu Service only through HTTP clients.

## 5. Dependencies and dk-common integration

### 5.1. Important rule

dk-common is a separate installable Python package.

dk-common can have its own virtualenv for developing dk-common itself.
However, fulfillment-service must not import from the dk-common virtualenv.

fulfillment-service has its own virtualenv and installs dk-common as a normal path dependency.

Forbidden:

```python
import sys
sys.path.append("../../libs/python/dk-common")
```

Forbidden:

```text
relative imports from libs using ../../../
```

Correct imports after installing dk-common:

```python
from dk_common.logging import configure_logging
from dk_common.correlation import CorrelationIdMiddleware
from dk_common.health import build_health_response
from dk_common.settings import BaseServiceSettings
```

### 5.3. Dependency ownership rule

Do not duplicate dk-common internal dependencies unless fulfillment-service imports them directly.

Rule:

```text
If a dependency is used only inside dk-common,
declare it only in libs/python/dk-common/pyproject.toml.

If fulfillment-service imports the dependency directly,
declare it in services/fulfillment-service/pyproject.toml.
```

Examples:

```text
fulfillment-service directly uses FastAPI, SQLAlchemy, Alembic, asyncpg, httpx.
So they must be in fulfillment-service pyproject.toml.

fulfillment-service should not directly use Loguru if it only calls configure_logging().
So Loguru can remain a dk-common dependency.
```

## 7. FastAPI app

Create app/main.py.

The app must:

```text
1. Load settings.
2. Configure logging through dk-common.
3. Add CorrelationIdMiddleware.
4. Add /health.
5. Include order routes.
```

Example:

```python
from fastapi import FastAPI

from dk_common.correlation import CorrelationIdMiddleware
from dk_common.health import build_health_response
from dk_common.logging import configure_logging

from app.api.routes import orders
from app.core.config import get_settings


def create_app() -> FastAPI:
    settings = get_settings()

    configure_logging(
        service_name=settings.service_name,
        environment=settings.environment,
        log_level=settings.log_level,
        log_format=settings.log_format,
    )

    app = FastAPI(title="Fulfillment Service", version="0.1.0")

    app.add_middleware(CorrelationIdMiddleware)

    @app.get("/health")
    async def health():
        return build_health_response(
            service_name=settings.service_name,
            environment=settings.environment,
            version="0.1.0",
        )

    app.include_router(orders.router)

    return app


app = create_app()
```

If dk-common already provides create_health_router(), it is acceptable to use it.
If not, use build_health_response() as shown above.

## 8. Database

### 8.1. PostgreSQL ownership

Fulfillment Service owns its own tables:

```text
orders
order_items
kitchen_tasks
task_dependencies
```

Even if all services use one physical PostgreSQL instance, Fulfillment Service must not read or write tables owned by Kitchen Service or Menu Service.

### 8.2. Async SQLAlchemy session

Create app/db/session.py:

```python
from collections.abc import AsyncGenerator

from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine

from app.core.config import get_settings

settings = get_settings()

engine = create_async_engine(settings.database_url, echo=False)

AsyncSessionLocal = async_sessionmaker(
    engine,
    expire_on_commit=False,
    class_=AsyncSession,
)


async def get_db_session() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionLocal() as session:
        yield session
```

### 8.3. Declarative base

Create app/db/base.py:

```python
from sqlalchemy.orm import DeclarativeBase


class Base(DeclarativeBase):
    pass
```

All SQLAlchemy models must inherit from Base.

## 9. Data model

### 9.1. orders

Purpose: stores order state owned by Fulfillment Service.

Fields:

```text
id                 UUID primary key
kitchen_id          UUID, required
status             string, required
pickup_deadline    timestamp with timezone, nullable
created_at         timestamp with timezone
updated_at         timestamp with timezone
```

Allowed statuses:

```text
created
accepted
scheduled
cooking
assembling
ready_for_pickup
handed_off
delayed
cancelled
failed
```

For this stage:

```text
POST /orders should create order.status = created
```

Do not implement full lifecycle transitions in this stage.

### 9.2. order_items

Purpose: stores original items requested in the order.

Fields:

```text
id             UUID primary key
order_id       UUID, required, foreign key to orders.id
menu_item_id   UUID, required
quantity       integer, required
created_at     timestamp with timezone
updated_at     timestamp with timezone
```

Constraints:

```text
quantity > 0
```

Recommended indexes:

```text
INDEX(order_id)
INDEX(menu_item_id)
```

Important:

```text
menu_item_id is an external reference to Menu Service.
Do not use a database foreign key to Menu Service tables.
```

### 9.3. kitchen_tasks

Purpose: stores global kitchen task state.

Fields from the original system model:

```text
id                            UUID primary key
order_id                      UUID, required, foreign key to orders.id
menu_item_id                  UUID, required
station_type                  string, required
station_id                    UUID, nullable
kds_task_id                   UUID, nullable
operation                     string, required
status                        string, required
estimated_duration_seconds    integer, required
displayed_at                  timestamp with timezone, nullable
started_at                    timestamp with timezone, nullable
sla_deadline_at               timestamp with timezone, nullable
completed_at                  timestamp with timezone, nullable
actual_duration_seconds       integer, nullable
delay_seconds                 integer, nullable
attempts                      integer, required, default 0
created_at                    timestamp with timezone
updated_at                    timestamp with timezone
```

Recommended extra fields for this stage:

```text
order_item_id                 UUID, nullable or required, foreign key to order_items.id
recipe_step_order             integer, required
item_unit_index               integer, required
```

Reason for the recommended extra fields:

```text
For quantity > 1, one order item can produce multiple task chains.
item_unit_index identifies which unit of the item the task belongs to.
recipe_step_order helps reconstruct dependencies and debug task generation.
```

If you want to stay strictly minimal, you can omit these extra fields.
However, keeping them makes tests and debugging much easier.

Allowed task statuses:

```text
created
queued
displayed
in_progress
done
failed
retrying
cancelled
```

For this stage:

```text
kitchen_tasks.status = created
attempts = 0
station_id = null
kds_task_id = null
displayed_at = null
started_at = null
completed_at = null
actual_duration_seconds = null
delay_seconds = null
```

Do not set status = queued yet.

Queued status belongs to Stage 5 when Redis Streams publishing is implemented.

Constraints:

```text
estimated_duration_seconds > 0
attempts >= 0
recipe_step_order > 0
item_unit_index > 0
```

Recommended indexes:

```text
INDEX(order_id)
INDEX(menu_item_id)
INDEX(status)
INDEX(station_type)
INDEX(order_item_id)
```

### 9.4. task_dependencies

Purpose: stores dependency graph between kitchen tasks.

Fields:

```text
task_id                  UUID, required, foreign key to kitchen_tasks.id
depends_on_task_id       UUID, required, foreign key to kitchen_tasks.id
created_at               timestamp with timezone
```

Primary key or unique constraint:

```text
UNIQUE(task_id, depends_on_task_id)
```

Recommended constraints:

```text
task_id != depends_on_task_id
```

Dependency rule for this stage:

```text
For each menu item unit, recipe steps form a linear chain by step_order.

Example:
  step 1 grill
  step 2 packaging

Then:
  packaging_task depends on grill_task
```

If recipe has 3 steps:

```text
step 2 depends on step 1
step 3 depends on step 2
```

Do not create dependencies between different order items or different quantity units.


Use httpx AsyncClient.

Clients must:

```text
3. Propagate X-Correlation-ID and X-Request-ID headers if available.
```

Do not call external services directly from route handlers.

## 12. API endpoints

### 12.1. POST /orders

Create order and build kitchen tasks.

Request:

```json
{
  "kitchen_id": "d53b7d88-b23c-4bb8-a403-6238c810092a",
  "pickup_deadline": "2026-04-30T18:45:00Z",
  "items": [
    {
      "menu_item_id": "3b675b7e-d4e1-4fc5-80dc-5f3ef89d55ce",
      "quantity": 2
    }
  ]
}
```

Validation:

```text
kitchen_id required
items required
items length > 0
quantity > 0
pickup_deadline optional
```

Processing:

```text
1. Validate request shape.
2. Call Kitchen Service: GET /kitchens/{kitchen_id}.
3. Require kitchen.status = active.
4. Call Menu Service: GET /kitchens/{kitchen_id}/menu.
5. Validate every requested menu_item_id is available and active for that kitchen.
6. For each requested menu_item_id, call Menu Service:
   GET /menu-items/{menu_item_id}/recipe.
7. Require recipe has at least one step.
8. Create order with status = created.
9. Create order_items.
10. For each order item quantity unit, create kitchen_tasks from recipe steps.
11. Set each kitchen_task.status = created.
12. Create task_dependencies between consecutive recipe steps for the same quantity unit.
13. Commit transaction.
14. Return created order summary.
```

Response: 201 Created

```json
{
  "id": "14cb4b0a-0100-4c6d-bb1f-6ce78ea45c8e",
  "kitchen_id": "d53b7d88-b23c-4bb8-a403-6238c810092a",
  "status": "created",
  "pickup_deadline": "2026-04-30T18:45:00Z",
  "items": [
    {
      "id": "2f451da5-c8d7-4053-814f-d6994da6f912",
      "menu_item_id": "3b675b7e-d4e1-4fc5-80dc-5f3ef89d55ce",
      "quantity": 2
    }
  ],
  "tasks_count": 4,
  "created_at": "2026-04-30T10:00:00Z",
  "updated_at": "2026-04-30T10:00:00Z"
}
```

Example:
If quantity = 2 and recipe has 2 steps, then tasks_count = 4.

Error cases:

```http
404 Not Found
```

```json
{
  "error": "kitchen_not_found",
  "message": "Kitchen not found"
}
```

```http

```

```json
{

}
```


## 13. Task generation algorithm

Implement task generation in a pure function or isolated service.

Suggested input:

```text
order_id
kitchen_id
order_items
recipes by menu_item_id
```

Suggested output:

```text
kitchen_tasks
task_dependencies
```

Algorithm:

```text
For each order item:
  For unit_index from 1 to quantity:
    Sort recipe steps by step_order ASC.
    For each recipe step:
      Create one kitchen_task:
        order_id = order.id
        menu_item_id = order_item.menu_item_id
        station_type = recipe_step.station_type
        operation = recipe_step.operation
        estimated_duration_seconds = recipe_step.duration_seconds
        status = created
        attempts = 0
        recipe_step_order = recipe_step.step_order
        item_unit_index = unit_index

    For each pair of consecutive steps:
      Create task_dependency:
        task_id = current_step_task.id
        depends_on_task_id = previous_step_task.id
```

Example:

```text
Order:
  Burger x 2

Recipe:
  step 1 grill cook_patty
  step 2 packaging pack_burger

Expected tasks:
  unit 1, step 1: grill
  unit 1, step 2: packaging
  unit 2, step 1: grill
  unit 2, step 2: packaging

Expected dependencies:
  unit 1 packaging depends on unit 1 grill
  unit 2 packaging depends on unit 2 grill

No dependency:
  unit 2 grill must not depend on unit 1 packaging
```

Important:

```text
Recipe steps from Menu Service may arrive unordered.
Fulfillment Service must sort by step_order ASC before building tasks.
```

## 17. Transactions and consistency

Order creation must be transactional for local database writes.

Required behavior:

```text
If validation against Kitchen Service or Menu Service fails:
  do not create order
  do not create order_items
  do not create kitchen_tasks

If local DB write fails:
  rollback transaction

If all external validation succeeds and DB writes succeed:
  commit order, order_items, kitchen_tasks, task_dependencies together
```

Because there is no Redis publishing in this stage, there is no outbox or distributed transaction yet.

Do not call Redis after commit in this stage.
