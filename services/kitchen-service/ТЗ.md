# ТЗ для агента: Этап 2 — Kitchen Service без KDS

## Контекст

Проект — polyglot monorepo для микросервисной системы dark kitchen.

На предыдущем этапе должна быть реализована общая Python-библиотека:

```text
libs/python/dk-common/
```

Она используется сервисами как installable package через Poetry path dependency и предоставляет общую техническую инфраструктуру:

```text
settings
logging
correlation middleware
health helper
infrastructure errors
```

На этом этапе нужно реализовать первый полноценный Python-сервис:

```text
services/kitchen-service/
```

Этап называется:

```text
21.3. Этап 2 — Kitchen Service без KDS
```

Цель этапа — реализовать управление кухнями и станциями.

---

## Главная цель этапа

Создать FastAPI-сервис `kitchen-service`, который умеет:

1. подключаться к PostgreSQL;
2. управлять таблицами `kitchens` и `stations`;
3. выполнять миграции через Alembic;
4. создавать кухни;
5. получать список кухонь;
6. получать кухню по id;
7. создавать станции внутри кухни;
8. получать станции кухни;
9. фильтровать станции по `station_type`;
10. обновлять `capacity` станции;
11. обновлять `status` станции;
12. использовать `dk-common` для настроек, логирования, correlation id и healthcheck;
13. иметь базовые тесты.

---

## Важное ограничение scope

На этом этапе Kitchen Service реализуется **без KDS**.

Не реализовывать:

```text
kds_station_tasks
GET /kds/stations/{station_id}/tasks
POST /kds/stations/{station_id}/tasks/{task_id}/claim
POST /kds/stations/{station_id}/tasks/{task_id}/complete
POST /kds/stations/{station_id}/tasks/{task_id}/fail
GET /internal/kds/dispatch-candidates
POST /internal/kds/stations/{station_id}/tasks
busy slot claiming logic
double claim protection
idempotent KDS delivery
callbacks в Fulfillment Service
Redis
MongoDB events
Prometheus business metrics
```

В этом этапе можно хранить поле `busy_slots` в таблице `stations`, но оно должно оставаться базовым числовым полем с default `0`. Логику claim/complete и атомарное увеличение `busy_slots` не делать до этапов KDS.

## Зависимости и `dk-common`

### Главное правило

`dk-common` имеет свой `pyproject.toml` и может иметь свой virtualenv для разработки самой библиотеки.

Но `kitchen-service` **не должен импортировать код из virtualenv библиотеки**.

`kitchen-service` должен устанавливать `dk-common` внутрь своего собственного virtualenv как обычную path dependency.

---

## `services/kitchen-service/pyproject.toml`

В `pyproject.toml` сервиса нужно явно указать собственные зависимости сервиса и path dependency на `dk-common`.

Пример:

```toml
[tool.poetry]
name = "kitchen-service"
version = "0.1.0"
description = "Kitchen Service for dark kitchen project"
authors = ["dark-kitchen"]
packages = [{ include = "app" }]

[tool.poetry.dependencies]
python = "^3.11"
fastapi = "^0.115.0"
uvicorn = { extras = ["standard"], version = "^0.30.0" }
pydantic = "^2.0"
pydantic-settings = "^2.0"
sqlalchemy = "^2.0"
alembic = "^1.13"
asyncpg = "^0.29"
dk-common = { path = "../../libs/python/dk-common", develop = true }

[tool.poetry.group.dev.dependencies]
pytest = "^8.0"
pytest-asyncio = "^0.23"
httpx = "^0.27"

[build-system]
requires = ["poetry-core"]
build-backend = "poetry.core.masonry.api"
```

### Как должны работать зависимости

При выполнении:

```bash
cd services/kitchen-service
poetry install
```

Poetry должен установить в virtualenv `kitchen-service`:

```text
1. зависимости самого kitchen-service;
2. пакет dk-common из ../../libs/python/dk-common;
3. транзитивные зависимости dk-common.
```

Сервису не нужно дублировать зависимости `dk-common`, если он не использует их напрямую.

Правило:

```text
Если зависимость используется только внутри dk-common,
она должна быть объявлена только в libs/python/dk-common/pyproject.toml.

Если зависимость напрямую импортируется кодом kitchen-service,
она должна быть объявлена в services/kitchen-service/pyproject.toml.
```

Например, если сервис напрямую импортирует FastAPI, SQLAlchemy, Alembic, asyncpg — эти зависимости должны быть в `services/kitchen-service/pyproject.toml`.

Если сервис только вызывает:

```python
from dk_common.logging import configure_logging
```

то зависимость `loguru` может оставаться только внутри `dk-common`, если код сервиса напрямую не импортирует `loguru`.

## Запрещённый способ подключения `dk-common`

Не использовать:

```python
import sys
sys.path.append("../../libs/python/dk-common")
```

Не использовать относительные импорты вида:

```python
from ../../../libs.python.dk_common import configure_logging
```

Правильный импорт после установки path dependency:

```python
from dk_common.logging import configure_logging
from dk_common.settings import BaseServiceSettings
from dk_common.correlation import CorrelationIdMiddleware
from dk_common.health import build_health_response
```

## Docker-сборка и `dk-common`

Docker build context должен быть корнем репозитория, чтобы Dockerfile видел и сервис, и общую библиотеку.

Пример compose build config:

```yaml
services:
  kitchen-service:
    build:
      context: ../..
      dockerfile: services/kitchen-service/Dockerfile
```

Пример Dockerfile:

```dockerfile
FROM python:3.11-slim

ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

WORKDIR /repo

RUN pip install --no-cache-dir uv

COPY libs/python/dk-common /repo/libs/python/dk-common
COPY services/kitchen-service /repo/services/kitchen-service

RUN uv pip install --system /repo/libs/python/dk-common
RUN uv pip install --system /repo/services/kitchen-service

WORKDIR /repo/services/kitchen-service

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

Важно: `uv pip install --system /repo/services/kitchen-service` должен видеть path dependency на `../../libs/python/dk-common`, поэтому структура внутри контейнера должна сохранять относительное расположение:

```text
/repo/libs/python/dk-common
/repo/services/kitchen-service
```

# 2. FastAPI-приложение

## `app/main.py`

Создать FastAPI app.

Требования:

1. подключить `configure_logging()` из `dk-common`;
2. подключить `CorrelationIdMiddleware` из `dk-common`;
3. добавить `/health` через helper из `dk-common`;
4. подключить роуты Kitchen Service;
5. не создавать таблицы через `Base.metadata.create_all()` в runtime — таблицы создаются миграциями Alembic.

Пример ожидаемой логики:

```python
from fastapi import FastAPI
from dk_common.correlation import CorrelationIdMiddleware
from dk_common.logging import configure_logging
from dk_common.health import build_health_response

from app.config import get_settings
from app.api.routes import router as api_router

settings = get_settings()

configure_logging(
    service_name=settings.service_name,
    environment=settings.environment,
    log_level=settings.log_level,
    log_format=settings.log_format,
)

app = FastAPI(title="Kitchen Service")
app.add_middleware(CorrelationIdMiddleware)


@app.get("/health")
async def health():
    return build_health_response(
        service_name=settings.service_name,
        environment=settings.environment,
        version="0.1.0",
    )


app.include_router(api_router)
```

Если в `dk-common` реализован `create_health_router()`, можно использовать его вместо ручного endpoint. Но FastAPI не должен быть runtime-зависимостью `dk-common`, если на этапе 1 это было исправлено.

# 3. PostgreSQL и SQLAlchemy

## Подход

Использовать SQLAlchemy 2.x async API и `asyncpg`.

## `app/db.py`

Реализовать:

```python
from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine
from sqlalchemy.orm import DeclarativeBase

from app.config import get_settings


class Base(DeclarativeBase):
    pass


settings = get_settings()

engine = create_async_engine(settings.database_url, echo=False)
AsyncSessionLocal = async_sessionmaker(
    engine,
    expire_on_commit=False,
    class_=AsyncSession,
)


async def get_db_session():
    async with AsyncSessionLocal() as session:
        yield session
```

Не использовать глобальную sync session, если выбран async stack.

# 5. Alembic

## Задача

Настроить Alembic внутри `services/kitchen-service`.

Миграции должны принадлежать конкретному сервису, а не `dk-common`.

`dk-common` не должен содержать Alembic env, SQLAlchemy-модели или миграции.

---

## Требования к Alembic

Создать:

```text
services/kitchen-service/alembic.ini
services/kitchen-service/alembic/env.py
services/kitchen-service/alembic/versions/<revision>_create_kitchens_and_stations.py
```

`alembic/env.py` должен импортировать metadata из `app.db.Base` и модели из `app.models`, чтобы autogenerate видел таблицы.

Пример:

```python
from app.config import get_settings
from app.db import Base
import app.models  # noqa: F401

target_metadata = Base.metadata
```

Так как используется async SQLAlchemy URL `postgresql+asyncpg://...`, в Alembic можно использовать один из двух вариантов:

### Вариант A — async Alembic env

Настроить `env.py` по async-шаблону Alembic с `async_engine_from_config`.

### Вариант B — sync URL для миграций

В `env.py` конвертировать URL:

```python
postgresql+asyncpg://...
```

в:

```python
postgresql://...
```

и использовать sync engine.

Если выбран вариант B, нужно добавить runtime dependency:

```text
psycopg[binary]
```

Для простоты этапа рекомендуется **вариант A**: async Alembic env, чтобы не добавлять второй PostgreSQL driver.

# 8. Ошибки

Создать `app/errors.py` или использовать локальные exception classes.

Минимальные ошибки:

```text
kitchen_not_found
station_not_found
```

Формат ошибок должен быть стабильным:

```json
{
  "error": "kitchen_not_found",
  "message": "Kitchen not found"
}
```

Можно использовать `DKCommonError` из `dk-common` как базовый инфраструктурный стиль, но доменные ошибки `kitchen_not_found` и `station_not_found` должны жить в `kitchen-service`, а не в `dk-common`.

Важно: `dk-common` не должен знать, что такое kitchen или station.

# 10. Тесты

Добавить базовые тесты.

## Рекомендуемый уровень

Для этого этапа достаточно component-style тестов FastAPI app + test database.

Возможные подходы:

1. использовать отдельную тестовую PostgreSQL БД из docker-compose;
2. использовать SQLite только для быстрых тестов, если код не завязан на PostgreSQL-specific features;
3. использовать Testcontainers позже.

Так как проект целится в PostgreSQL + Alembic, предпочтительный путь — тестовая PostgreSQL БД.

---

## `tests/conftest.py`

Подготовить:

```text
- test settings;
- async test db session;
- применение миграций или создание схемы для тестов;
- FastAPI AsyncClient;
```

# 13. Docker Compose

Если в репозитории уже есть `deploy/compose/docker-compose.yml`, добавить туда `kitchen-service`.

Минимальная конфигурация:

```yaml
services:
  kitchen-service:
    build:
      context: ../..
      dockerfile: services/kitchen-service/Dockerfile
    environment:
      SERVICE_NAME: kitchen-service
      ENVIRONMENT: local
      LOG_LEVEL: INFO
      LOG_FORMAT: json
      DATABASE_URL: postgresql+asyncpg://dark_kitchen:dark_kitchen@postgres:5432/kitchen_service
```

Если PostgreSQL уже один общий физический контейнер для MVP, можно использовать отдельную БД или отдельную схему для `kitchen-service`.


Допустимо для MVP:

```text

```

Сервисы не должны читать и писать чужие таблицы напрямую.

# 14. Что не переносить в `dk-common`

Не добавлять в `dk-common`:

```text
Kitchen model
Station model
SQLAlchemy Base для конкретного сервиса
Alembic env.py
Alembic migrations
KitchenRepository
StationRepository
KitchenCreate / StationCreate schemas
business statuses kitchen/station
endpoint logic
```

`dk-common` может использоваться только для:

```text
BaseServiceSettings
configure_logging
CorrelationIdMiddleware
build_health_response / create_health_router
базовых инфраструктурных ошибок
```

# 15. Definition of Done

Этап считается завершенным, если:

1. существует installable сервис `services/kitchen-service`;
2. сервис устанавливается через `poetry install`;
3. `dk-common` подключен через Poetry path dependency:

```toml
dk-common = { path = "../../libs/python/dk-common", develop = true }
```

4. сервис импортирует `dk-common` обычным импортом:

```python
from dk_common.logging import configure_logging
```

5. нет `sys.path.append` и относительных импортов в `libs`;
6. Alembic настроен внутри `services/kitchen-service`;
7. миграция создает таблицы `kitchens` и `stations`;
8. миграция не создает `kds_station_tasks`;
9. `GET /health` работает;
10. `POST /kitchens` создает кухню;
11. `GET /kitchens` возвращает список кухонь;
12. `GET /kitchens/{kitchen_id}` возвращает кухню по id;
13. `POST /kitchens/{kitchen_id}/stations` создает станцию;
14. `GET /kitchens/{kitchen_id}/stations` возвращает станции кухни;
15. `GET /kitchens/{kitchen_id}/stations?station_type=grill` фильтрует станции;
16. `PATCH /stations/{station_id}/capacity` обновляет capacity;
17. `PATCH /stations/{station_id}/status` обновляет status;
18. `capacity` сохраняется;
19. `visible_backlog_limit` сохраняется;
20. `busy_slots` при создании станции равен `0`;
21. базовые тесты проходят;
22. в сервисе нет KDS endpoints и KDS таблиц.
